# Cursor 使用指南（Step-by-Step）

以 **Cursor** 为主要工具，从零到上手的 AI 辅助编程操作指南。

---

## 一、AI 辅助编程能做什么

- **写新代码**：按需求描述生成函数、模块、小项目骨架。
- **改现有代码**：重命名、重构、加注释、修 bug、换写法（如换成 Rust）。
- **读代码**：解释某段逻辑、总结文件/项目结构、画调用关系。
- **补全与建议**：行内补全（Tab）、多行生成、根据注释补实现。
- **写测试与文档**：生成单元测试、README、API 说明。

用好 Cursor，可以把「查文档、搜示例、试错」的时间压缩，把精力放在设计和验证上。

---

## 二、Cursor 快速准备（Step 0）

### 2.1 安装与登录

1. 打开 [cursor.com](https://cursor.com)，下载对应系统版本并安装。
2. 首次打开用 GitHub / Google 登录，按提示完成账号绑定。
3. 建议在 **Settings → Cursor Settings** 里开启「Beta」相关能力（如 Composer、Agent 等），以便用最新功能。

### 2.2 打开你的项目

- **File → Open Folder** 打开项目根目录（例如你的 `linbozhang.github.io`）。
- Cursor 会索引整个工程，侧边栏会显示文件树；索引完成后，AI 才能更好理解「当前项目」。

### 2.3 分清两个入口：Chat 与 Composer

| 入口 | 快捷键 | 适合做什么 |
|------|--------|------------|
| **Chat** | `Ctrl+L`（Win）/ `Cmd+L`（Mac） | 问答、解释代码、单文件小改、写片段 |
| **Composer** | `Ctrl+I`（Win）/ `Cmd+I`（Mac） | 多文件编辑、按步骤实现需求、大范围重构 |

- 小问题、问「这段代码什么意思」→ 用 **Chat**。
- 「帮我加一个页面并改导航」→ 用 **Composer**。

## 三、Step-by-Step：用 Cursor 做一次完整需求

下面用「在 mdBook 里加一个新章节并写一点内容」为例，走一遍标准流程。你可以把「需求」换成你自己的（例如加接口、修 bug、写脚本）。

### Step 1：把需求说清楚（写 Prompt）

在 **Composer** 里输入时，尽量包含：

- **做什么**：例如「在 AI - 代码生成 下面加一个子章节：Cursor 入门」。
- **放在哪**：例如「在 `src/ai/coding/` 下新建 `xxx.md`，并在 `SUMMARY.md` 里对应位置加一条」。
- **约束**：例如「用中文、不要动其他已有章节」。

示例 Prompt：

```text
在 mdBook 的「AI - 代码生成」下增加一个子章节「Cursor 入门」：
1. 在 src/ai/coding/ 下新建 cursor-intro.md，写一段简短介绍（3～5 句）。
2. 在 src/SUMMARY.md 里，在「代码生成」下面增加一条指向 cursor-intro.md 的目录项。
不要改其他文件。
```

要点：**一次只做一件事、写清「改哪 / 加哪」**，AI 更容易执行正确。

### Step 2：让 AI 看到相关文件（@ 引用）

在输入框里用 `@` 引用文件或符号，避免 AI 乱猜路径：

- `@SUMMARY.md`：让 AI 知道目录结构。
- `@src/ai/coding/README.md`：让 AI 知道「代码生成」章节的写法风格。
- `@src/ai/`：引用整个目录，适合「在这一类文件里做类似修改」。

例如在 Composer 里写：

```text
@src/SUMMARY.md @src/ai/coding/README.md
在「代码生成」下增加子章节「Cursor 入门」，新建 src/ai/coding/cursor-intro.md，并在 SUMMARY 里加一条链接。
```

这样 AI 会基于你引用的文件来改，减少改错文件或漏改。

### Step 3：执行并检查 Diff

1. 发送消息后，Cursor 会生成修改方案，并在界面上用 **Diff** 标出每个文件的改动。
2. **逐个文件点开**，看：
   - 是否只改了该改的地方；
   - 链接路径、文件名是否一致（如 `SUMMARY.md` 里的路径要和实际文件对应）。
3. 若某处不对，可以在 Composer 里继续说：「只改 SUMMARY 里那一行的路径为 xxx」或「cursor-intro.md 第二段改成 xxx」。
4. 确认无误后，点击 **Accept**（或按提示接受部分改动）。

不要不检查就全部接受，尤其是动到配置、路由、目录结构时。

### Step 4：本地验证

- 对 mdBook：在项目根目录执行 `mdbook serve`，浏览器里打开对应章节，确认：
  - 新页面能打开；
  - 侧栏里「Cursor 入门」在「代码生成」下面且可点击。
- 对代码项目：编译/跑测试/跑一遍关键流程。

若验证失败，把报错或现象贴回 Chat/Composer，让 AI 修（例如「SUMMARY 里写错了，应该是 `ai/coding/cursor-intro.md`」）。

### Step 5：迭代与细化

若第一次结果不理想，可以分步说：

- 「先只加 SUMMARY 里的那一行，不要新建文件。」
- 「cursor-intro.md 里再加一小节：如何用 @ 引用文件，约 2～3 句。」

把大需求拆成小步骤，每步验证，再继续下一步。

## 四、日常高频用法（按场景）

### 4.1 解释一段代码

1. 在编辑器里选中这段代码。
2. `Ctrl+L` 打开 Chat，输入：「解释一下这段代码在做什么，有哪些注意点。」
3. 若希望结合项目上下文，可在输入框里加 `@文件名` 或 `@文件夹`。

### 4.2 按注释补全实现

1. 在代码里先写清注释，例如：

   ```csharp
   // 遍历 list，把每个元素转成大写并去重，返回新列表
   ```

2. 换行后等 Cursor 的灰色补全，或 `Ctrl+L` 问：「按上面注释实现这个函数。」
3. 接受补全或 Chat 生成的代码后，跑一下测试或运行确认行为。

### 4.3 修 Bug：带报错和上下文

1. 把**报错信息**完整复制到 Chat（可带堆栈）。
2. 必要时 `@` 引用报错所在的文件甚至整个项目。
3. Prompt 示例：「运行 xxx 时出现下面的报错，请根据 @某文件 分析可能原因并给出修改建议。」

AI 会结合报错和代码给出修改方案，你再按 Step 3～4 检查并验证。

### 4.4 重构：重命名、拆函数、换风格

1. 在 Composer 里说清目标，例如：「把 `src/utils/helper.cs` 里超过 20 行的函数拆成 3 个小函数，保持对外接口不变。」
2. 用 `@src/utils/helper.cs` 引用该文件；若涉及调用方，可再 `@` 引用调用处或整个 `src/`。
3. 接受改动后运行测试，确保行为不变。

### 4.5 写测试

1. 选中要测的函数或类，或 `@` 引用该文件。
2. Chat/Composer：「为这个函数写单元测试，覆盖正常情况和边界情况，用 [你项目的测试框架，如 xUnit / pytest]。」
3. 把生成的测试文件放到项目约定目录，运行测试并补漏。

## 五、让 Cursor 更懂你的项目：Rules 与上下文

### 5.1 项目级规则（.cursor/rules 或 .cursorrules）

在项目根目录建 `.cursorrules`（或 `.cursor/rules` 下的规则文件），用自然语言写「这个项目的约定」，例如：

- 文档和注释用中文。
- 本仓库是 mdBook，新章节加在 `src/` 下，并在 `SUMMARY.md` 里登记。
- 代码风格：例如 C# 用 Allman 括号、Rust 用默认 rustfmt。

之后 AI 在回答和改代码时会尽量遵守这些规则。

### 5.2 用好 @ 的几种方式

- `@文件名`：指定某个文件。
- `@文件夹/`：指定目录（如 `@src/ai/`）。
- `@Codebase` 或 `@codebase`：让 AI 在整库里搜索（适合「在哪里用到了这个函数」）。
- `@Web`：允许 AI 查网络（版本号、最新 API 等），按需使用。

尽量把「和这次修改相关的文件」都 @ 进来，减少幻觉和误改。

## 六、注意事项与最佳实践

1. **先小范围再扩大**：先让 AI 改一个文件或一个函数，确认风格和逻辑没问题，再让它改更多。
2. **关键逻辑必看、必测**：算法、安全、数据一致性等，不要只看 AI 的总结，要自己过一遍代码并跑通。
3. **敏感信息**：不要贴密钥、密码、真实用户数据进 Chat/Composer；可贴脱敏后的配置或示例。
4. **版本与依赖**：若问「怎么用某某库」，说明你的语言和版本（如「C# 12 / .NET 8」），避免过时或错误的写法。
5. **迭代时收窄范围**：若 AI 某次改多了，下次就说「只改 xxx 文件里的 yyy 部分，其它不要动」。

---

## 七、小结：一次请求的检查清单

- [ ] 需求是否写清：做什么、改哪/加哪、有无约束？
- [ ] 是否用 `@` 引用了相关文件或目录？
- [ ] 用的是 Chat（小改/问答）还是 Composer（多文件/大改）？
- [ ] 接受前是否看了 Diff 并做了本地验证？
- [ ] 项目约定是否写在 .cursorrules 里（可选但推荐）？

按上述步骤在 Cursor 里多练几次，就能稳定地用 AI 辅助完成「代码生成、修改、解释、测试」等任务。
