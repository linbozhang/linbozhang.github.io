# Cursor Rules 与 Skills 的区别

虽然 Rules 和 Skills 都是对 AI Agent 行为规范的描述，但 Cursor 将它们分开设计，是因为它们的**设计目的、触发机制与使用场景**不同。本文说明两者的区别与何时用哪个。

---

## 1. 核心区别（一句话）

- **Rules**：**约束与规范**（「不能做什么」「必须怎么做」「在什么情况下遵守什么约定」）。
- **Skills**：**能力与工作流**（「如何完成某个任务」「按什么步骤执行」「需要什么工具或知识」）。

---

## 2. 设计目的对比

| 维度 | Rules | Skills |
|------|-------|--------|
| **本质** | 约束条件、规范、约定 | 可执行的工作流、任务能力 |
| **作用** | 限制或引导 Agent 的**回答方式**（如代码风格、命名规范、必须同步更新 SUMMARY） | 教 Agent **如何完成一个任务**（如按模板整理文档、生成 commit message、执行某流程） |
| **类比** | 「项目规范文档」「代码审查清单」 | 「操作手册」「工作流程指南」 |

---

## 3. 触发机制对比

| 维度 | Rules | Skills |
|------|-------|--------|
| **何时生效** | 由 **`globs`**（文件路径匹配）或 **`alwaysApply: true`** 决定 | 由 Agent 根据 **`description`** 与**用户请求的语义匹配**自动判断 |
| **触发依据** | 当前打开的文件路径 | 用户消息内容（如「按模板整理」「生成 commit message」） |
| **示例** | 打开 `src/**/*.md` 时自动加载「文档库结构约定」 | 用户说「按模板在文档库加一篇文档」时，Agent 发现 description 匹配，启用 `doc-library` Skill |

---

## 4. 结构与组织方式对比

| 维度 | Rules | Skills |
|------|-------|--------|
| **文件结构** | **单文件**：`.cursor/rules/xxx.mdc` | **目录结构**：`.cursor/skills/<name>/SKILL.md` + 可选 `reference.md`、`examples.md`、`scripts/` |
| **配置方式** | YAML frontmatter：`description`、`globs`、`alwaysApply` | YAML frontmatter：`name`、`description`（无 globs/alwaysApply） |
| **内容组织** | 通常较短（建议 50 行内），一条规则一个文件 | 可更长（建议 SKILL.md 500 行内），支持渐进披露（链接到 reference.md 等） |

---

## 5. 使用场景对比

### 适合用 Rules 的场景

- **代码风格约定**：TypeScript 错误处理必须记录日志、禁止空 catch。
- **项目结构约定**：编辑 `src/` 下 `.md` 时必须同步更新 SUMMARY。
- **全局行为约束**：每轮对话必须以提问结束（`alwaysApply: true`）。
- **文件类型特定规范**：编辑 `**/*.ts` 时遵守的 TypeScript 约定。

### 适合用 Skills 的场景

- **多步骤工作流**：按模板在文档库新增文档（确定归属 → 定路径 → 写内容 → 更新 SUMMARY → 自检）。
- **需要领域知识**：如何用 pdfplumber 提取 PDF 文本、如何生成符合规范的 commit message。
- **可复用任务**：代码审查、Excel 分析、数据库查询等。
- **需要工具/脚本**：Skill 目录下可放 `scripts/`，Rules 不支持。

---

## 6. 为什么分开设计？

### 6.1 职责分离

- **Rules**：负责「约束」——在 Agent 回答时注入「必须遵守的规范」，减少违反约定的输出。
- **Skills**：负责「能力」——在 Agent 需要完成某任务时提供「如何做的步骤与知识」，让 Agent 能执行复杂工作流。

两者配合：Rules 确保输出符合规范，Skills 提供执行能力。

### 6.2 触发机制不同

- **Rules**：基于**文件路径**或**全局**（`globs` / `alwaysApply`），适合「编辑某类文件时自动约束」。
- **Skills**：基于**语义匹配**（用户请求与 `description`），适合「用户提出任务时自动启用」。

若合并，难以同时满足「按文件触发」和「按语义触发」的需求。

### 6.3 内容组织需求不同

- **Rules**：通常简短、聚焦，一条规则一个文件，便于维护。
- **Skills**：可能包含多步骤、示例、参考文档、脚本，需要目录结构支持渐进披露。

---

## 7. 实际例子：文档库场景

### 用 Rule 实现

**`.cursor/rules/docs-structure.mdc`**：
- **作用**：约束——编辑 `src/**/*.md` 时提醒「必须同步更新 SUMMARY」。
- **触发**：打开 `src/` 下任意 `.md` 时自动加载。
- **内容**：简短几条约定（正文在 `src/`、SUMMARY 是唯一来源、路径规范等）。

### 用 Skill 实现

**`.cursor/skills/doc-library/SKILL.md`**：
- **作用**：能力——教 Agent「如何按模板在文档库新增/整理文档」。
- **触发**：用户说「按模板加一篇文档」「整理文档库」时，Agent 匹配 description 后启用。
- **内容**：完整流程（确定归属 → 定路径 → 写内容 → 更新 SUMMARY → 自检），并链接到 `template.md` 查看详细规范。

**配合使用**：
- 用户说「按模板在 AI 下加一篇文档」→ Agent 启用 `doc-library` Skill，按流程执行。
- 执行过程中编辑 `src/ai/xxx.md` → Rule `docs-structure` 自动生效，提醒同步更新 SUMMARY。

---

## 8. 小结

| 维度 | Rules | Skills |
|------|-------|--------|
| **本质** | 约束与规范 | 能力与工作流 |
| **触发** | 文件路径（globs）或全局（alwaysApply） | 语义匹配（description） |
| **结构** | 单文件 `.mdc` | 目录 + `SKILL.md` + 可选文件 |
| **长度** | 通常较短（建议 50 行内） | 可更长（SKILL.md 建议 500 行内） |
| **场景** | 代码风格、项目约定、必须遵守的规范 | 多步骤任务、工作流、需要领域知识的能力 |

**为什么分开**：两者职责不同（约束 vs 能力）、触发机制不同（文件路径 vs 语义）、组织需求不同（单文件 vs 目录结构）。分开设计能更清晰地表达意图，也便于维护与扩展。
